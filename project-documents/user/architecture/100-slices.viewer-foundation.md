# Slice Plan: Viewer Foundation

## Parent Document
[001-concept.migratory-viewer.md](001-concept.migratory-viewer.md) — Project concept for the migratory real-time visualization client.

## Foundation Work

1. [x] **(100) Project Scaffold and Rendering Core** — Vite + pnpm project setup (TypeScript, dev server, build config), Three.js scene initialization using `WebGPURenderer` (via `three/webgpu`, automatic WebGL 2 fallback), physically correct lighting, orthographic camera with pan/zoom controls, and a static test scene rendering N instanced cones at random positions over a flat ground plane. Render loop via `renderer.setAnimationLoop()` with `THREE.Timer`. No WebSocket, no simulation data — this slice proves the rendering pipeline works and establishes the project structure. Includes: `package.json`, `tsconfig.json` (with `moduleResolution: "bundler"`), Vite config, source directory layout, and a `dev` script that launches the viewer in a browser. The instanced mesh uses the same cone geometry and profile-based coloring approach proven in the `threejs_boid_terrain.html` artifact. GPU device loss handling is registered at the renderer level to pause and resume the render loop. Effort: 2/5

## Feature Slices

2. [x] **(101) WebSocket Consumer and Live Entity Rendering** — Connect to the world server's WebSocket endpoint, receive the initial snapshot (world bounds, entity state arrays), and render entities from server data instead of random positions. Per-tick state updates drive instanced mesh matrix updates in the render loop. Connection lifecycle: connect, receive snapshot, consume ticks, handle disconnect with reconnection and status indicator. The deserialization layer parses the binary wire protocol defined in migratory slice 306: `DataView` for fixed-size headers (type byte, tick number, entity count, world bounds), `Float64Array` views for positions and velocities, `Int32Array` view for profile indices. Little-endian throughout (all `DataView` reads pass `true` for `littleEndian`), row-major. The deserialization layer validates at parse boundaries: reject unknown message type bytes, verify buffer length matches claimed entity count before wrapping typed arrays, and reject entity counts exceeding a configurable upper bound. On validation failure, log and discard the frame without disconnecting. This slice establishes the `ViewerState` interface as the central shared state container — the connection handler is the sole writer; rendering and UI components read from it. Entity count is stable between snapshots in v1; if a STATE_UPDATE's entity count differs from the last SNAPSHOT, log a warning and reconnect to obtain a fresh snapshot. The WebSocket endpoint URL is configurable via environment variable or runtime parameter. This is the slice where the viewer becomes a real client. Dependencies: (100) Project Scaffold. External: migratory slices 305, 306 (both complete). Risk: Low. Effort: 3/5

3. [x] **(102) Terrain Rendering** — Consume the TERRAIN (0x03) message from the server and render the elevation grid as a displaced `PlaneGeometry` with computed vertex normals. The TERRAIN message is sent once per connection, after SNAPSHOT and before any STATE_UPDATE frames; the viewer parses the header (`rows`, `cols`, `resolution`, `origin_x`, `origin_y`), reads the row-major `float64` elevation array, and constructs a plane sized `cols × resolution` by `rows × resolution` positioned at `(origin_x, origin_y)` in world space. Vertex y-coordinates are set from elevation values (row 0 = southern edge, matching the server's `Grid` convention); normals are computed via `BufferGeometry.computeVertexNormals()` so lighting works correctly. Entities render above terrain height via a terrain height lookup (bilinear interpolation on the elevation grid); the lookup falls back to `y=0` if no TERRAIN message has arrived, preserving the current flat-ground behavior. A conforming client tolerates connections that never send TERRAIN (server's configured-default case per slice 507). The client-side WebSocket `maxPayload` / equivalent is raised to match the server's 32 MiB default (documented in the TERRAIN spec). Deserialization lives alongside the existing 0x01/0x02 parsers, dispatched by the leading type byte; validation rejects zero/negative `rows`/`cols`/`resolution` and verifies the buffer length matches `33 + rows × cols × 8` before wrapping the `Float64Array` view. No biome rendering in this slice. Dependencies: (101) WebSocket Consumer. External: migratory slice 507 (TERRAIN wire format). Risk: Low. Effort: 2/5

4. [ ] **(109) Biome Rendering** — Render biome classifications as a colored overlay atop terrain, driven by a future BIOME message analogous to TERRAIN (0x03). Each cell's biome id maps to a configurable color (grassland, forest, desert, water, etc.); rendering is either a per-vertex color attribute on the terrain mesh or a lookup texture sampled in a custom material, chosen when the protocol extension lands. Biome colors live in `config.ts` keyed by biome id. Entities are unaffected by biome beyond visual context. Dependencies: (102) Terrain Rendering. External: migratory slice 502 (biome layer) and its wire-protocol extension. Risk: Low. Effort: 2/5

5. [ ] **(103) Environment Overlay Rendering** — Render the remaining environment layers as visual overlays atop terrain: resource points as billboard sprites or small meshes with type-based icons, threat zones as translucent radial gradient meshes (red tint, configurable opacity), and stigmergic trails as a dynamic texture or decal layer updated per tick. Each layer is independently toggleable via a UI control. Dependencies: (102) Terrain Rendering, (109) Biome Rendering. External: migratory slices 503, 505, 506 (not started). Risk: Medium — trail rendering performance at scale. Effort: 3/5

6. [x] **(104) Camera Modes and Navigation** — Extend the camera system beyond the foundation's basic orthographic pan/zoom. Add: perspective mode toggle (smooth animated transition between orthographic top-down and perspective orbit), orbit controls in perspective mode (drag to rotate, scroll to zoom), follow-cam mode that tracks a selected entity or group centroid, minimap in corner showing full world bounds with viewport indicator. Dependencies: (100) Project Scaffold. Risk: Low. Effort: 2/5

7. [x] **(105) HUD and Status Panel** — Heads-up display with simulation metadata and controls. Connection status (connected/disconnected/reconnecting), tick rate and frame rate counters, entity count by profile type, simulation time (when server slice 308 provides temporal state), profile legend (color key mapping profile names to their rendering color). Overlay panel is minimal DOM positioned over the canvas, styled to match the simulation aesthetic. Toggle visibility with a hotkey. Dependencies: (101) WebSocket Consumer. Risk: Low. Effort: 2/5

8. [x] **(108) Camera Constraints and Pan** — Constrain the orthographic camera so the viewer maintains a consistent, world-grounded viewpoint, and add click-drag panning. Zoom-out is clamped to the level where the camera frustum exactly frames the world bounds (max zoom-out = world-fit); because only one center position is valid at that zoom level, fully zooming out naturally resets any pan with no special-case code. Left-click + drag pans the camera by translating its center in the XZ plane, with drag distance scaled so one screen pixel maps to one world unit at the current zoom. Pan is clamped so the camera frustum edges cannot cross world bounds. A single `allowOutOfBoundsView` config flag (default `false`) disables both the zoom and pan clamps as a debug escape hatch. Input is routed through a thin action-abstraction layer (e.g. `panStart` / `panMove` / `panEnd`) rather than wiring mouse events directly into camera logic, so rebinding to other inputs (middle-click, modifier keys, gamepad, macOS three-finger drag — which the OS delivers as normal mouse events) is a config/wiring change later rather than a rewrite. Snap pan only; no inertia or smoothing in this slice. Existing zoom-to-center wheel behavior is preserved. Camera parameters (clamp flag, pan speed) live in `config.ts` alongside existing camera/rendering settings. Note: this slice is a prerequisite subset of the future (104) Camera Modes and Navigation slice — record that relationship when 104 is picked up. Dependencies: (100) Project Scaffold. Risk: Low. Effort: 2/5

9. [x] **(110) Terrain Surface Material** — Replace the flat `MeshLambertMaterial` on the terrain mesh with a `MeshStandardNodeMaterial` (Three.js TSL / WebGPU node material) that blends surface appearance by slope. The world-space surface normal's Y-component (`normalWorld.y`) drives a `smoothstep` blend in the GPU shader: near-flat faces receive the "surface" appearance (vegetation, soil), near-vertical faces receive the "rock/cliff" appearance. The blend boundary and softness are configurable parameters. In this slice, both surface types are defined as solid colors with roughness/metalness values — no texture maps yet — but the material architecture is designed to accept texture nodes in slice 111. The biome config structure introduced here (`BiomeConfig`) lives in `config.ts` and defines surface color, cliff color, roughness, metalness, and slope-blend parameters as a swappable unit; changing biomes is a single config substitution. Lighting is upgraded from hemisphere+directional to a tuned PBR setup matching the concept art (warm key light, cool fill, adjusted intensities). The terrain material is rebuilt when biome config changes; no per-frame CPU work — all blending runs on the GPU. Scaling: one draw call, shader cost O(1) regardless of entity count or world size. Practical limit is terrain polygon count, already capped by `terrainMaxCells`. Dependencies: (102) Terrain Rendering. Risk: Low. Effort: 2/5

10. [x] **(111) Terrain Slab and Texture** — Add geological slab depth to the terrain and introduce texture maps to the surface material. The slab is four side-wall meshes plus a bottom face, sized to world bounds and slab depth (configurable in world units); side walls use a dedicated cliff material (dark rock, no vegetation blending). Texture maps (diffuse + normal) replace the solid colors from slice 110 — the `BiomeConfig` gains optional `surfaceTexture` and `cliffTexture` fields referencing loaded `THREE.Texture` assets; when absent the system falls back to solid color (preserving slice 110 behavior). Texture tiling scale is a `BiomeConfig` parameter so different biomes can scale texture density independently. Triplanar UV sampling (projecting texture from three axes and blending by normal) is used on the top surface to eliminate UV seams at steep transitions — this is a TSL node graph, not a CPU operation. Slab geometry is rebuilt on world-bounds change (same trigger as terrain mesh). Scaling: slab is 5 static meshes (4 walls + bottom), constant cost; texture memory is the binding constraint — a 2K diffuse + normal pair is ~32 MB, well within GPU budget for 1–4 biome textures loaded simultaneously. Dependencies: (110) Terrain Surface Material. Risk: Low–Medium — triplanar UV in TSL requires verification against Three.js WebGPU backend. Effort: 3/5

11. [x] **(112) Terrain Wire Protocol v2 (Chunked + Compressed)** — Upgrade the terrain ingest path to the server's versioned v2 wire format, replacing the slice 102 single-shot `0x03` parser with a dispatch state machine that handles both the v2 single-shot `TERRAIN` (`0x03`) and the chunked `TERRAIN_HEADER` (`0x05`) + `TERRAIN_CHUNK` (`0x04`) sequence. The v2 header carries a flags byte declaring dtype (`f32` / `f64` / `uint16`) and compression (`none` / `zstd` / `lz4`); for `uint16`, the 16 bytes following the flags carry `elevation_min` / `elevation_max` for dequantization to f64 via `min + (u16 / 65535) * (max - min)`. Compressed payloads are standard frames — zstd (RFC 8478, magic `0x28 B5 2F FD`) and LZ4 Frame format (magic `0x04 22 4D 18`); the viewer ships WASM/JS decoders for both (e.g. `fzstd` and `lz4js`) so server config can change algorithm without a viewer change. The chunked path maintains per-connection state (`IDLE` → `EXPECTING_CHUNKS` → `IDLE`), allocates a `Float64Array(rows * cols)` on `TERRAIN_HEADER`, writes each decoded chunk into the grid by `row_offset` / `col_offset` (not arrival order), and validates rectangular partition coverage on completion before handing the grid to the renderer. This slice is also our first instance of versioned protocol messages — establishes the convention that protocol errors (reserved-bit set, unknown dtype/compression, out-of-state chunk, length mismatch, decoder failure) close the WebSocket with code 1002 rather than attempting in-stream recovery. Includes a ground-truth unit test parsing the spec's worked example (2×2 f32 + zstd) end-to-end. Renderer integration is unchanged: the slice 102 `TerrainGrid` shape and the renderer-facing handoff remain f64 row-major. Dependencies: (102) Terrain Rendering. External: migratory v2 terrain wire spec (see slice 112 design). Risk: Medium — third-party decompressor bundle size and WASM init are the main unknowns. Effort: 3/5

12. [ ] **(113) Entity Pipeline Performance** — Address two concrete throughput bottlenecks identified at 10k entities / 250 server tps: (1) zero-copy deserialization — eliminate the `buffer.slice()` + intermediate `Float64Array` allocations in `parseStateUpdate` and `parseSnapshot` by wrapping views directly into the existing `ViewerState` buffers via `new Float64Array(buffer, offset, count)`, removing ~320 KB of allocation and copy per tick; (2) render-skip and baked height cache — track `lastRenderedTick` in the render loop and skip `updateEntities` when the tick hasn't advanced, plus pre-bake terrain heights into a `Float32Array` in `applyStateUpdate` so the per-entity bilinear interpolation loop runs once per tick rather than once per frame. No changes to the rendering visual output — same positions, same heights, same colors. Dependencies: (101) WebSocket Consumer, (102) Terrain Rendering. Risk: Low — mechanical buffer arithmetic changes with clear before/after correctness checks. Effort: 2/5

13. [ ] **(114) Entity Position dtype Negotiation (f32/f64)** — Update the viewer to consume a new per-message dtype flag byte in SNAPSHOT (offset 25) and STATE_UPDATE (offset 9) that declares whether entity positions and velocities are encoded as f32 (16 bytes per entity) or f64 (32 bytes per entity). Add a `PositionDtype` constant (`F64: 0x00`, `F32: 0x01`) in `protocol/types.ts` alongside `TerrainDtype`. In `deserialize.ts`: read the flag byte at the documented offset, reject unknown values with `console.warn` + return `null`, shift all subsequent field offsets by 1, branch on dtype to construct `Float32Array` or `Float64Array` and compute `expectedBytes` using the correct per-entity width (16 f32, 32 f64). Update `ParsedSnapshot` and `ParsedStateUpdate` interfaces to declare `positions: Float32Array | Float64Array` and `velocities: Float32Array | Float64Array`. Update `ViewerState` and `state.ts` `applyStateUpdate` to accept the union type. `rendering/entities.ts` passes the array directly to Three.js — `BufferAttribute` accepts both dtypes natively, so no rendering branch is needed. The f32 path halves per-tick bandwidth (~313 KB → ~156 KB at 10k entities); the f64 path preserves exact backward compatibility during the server transition. The server signals the dtype at startup via `ServerConfig.simulation.agent_wire_dtype`; the viewer never negotiates — it reads the flag and adapts per message. Dependencies: (101) WebSocket Consumer. External: migratory server f32 entity wire update (flag byte insertion). Risk: Low — dtype flag is a mechanical header change; union type flows through a small number of files. Effort: 2/5

## Integration Work

9. [ ] **(106) Performance Profiling and Optimization** — Profile rendering and deserialization performance at target entity counts (10K, 50K, 100K). Identify bottlenecks in the instanced mesh update path, WebSocket deserialization, and garbage collection. Implement optimizations as needed: reuse typed array buffers (avoid allocation per tick), frustum culling on the client side (skip matrix updates for off-screen instances), LOD reduction for distant entities (smaller geometry or point rendering beyond a threshold). Establish a performance budget: maintain 30fps at 50K entities on a mid-range GPU. Dependencies: (101) WebSocket Consumer. Risk: Medium — unknown bottleneck distribution. Effort: 3/5

## Notes

- **Slice 101 is the critical path.** Everything after it depends on live data. The foundation slice (100) can be built immediately. The wire protocol (306) is complete — no protocol uncertainty remains.
- **Slice 102 consumes the TERRAIN (0x03) message** defined in migratory slice 507. Biome rendering (109) and remaining environment overlays (103) are gated on further migratory protocol extensions for biome, resource, threat, and trail layers.
- **Slice 104 (camera) and 105 (HUD) are independent of each other** and of the environment rendering slices. They can be developed in parallel with 102/103/109 once 101 is complete.
- **No client-to-server commands in this plan.** The world server's v1 protocol is server-push only. Interactive controls (spawning entities, adjusting parameters) are future work gated on the server's "Administrative Client Commands" future slice.
- **TypeScript confirmed.** The binary protocol parsing — computing byte offsets into `ArrayBuffer`, wrapping sections as `Float64Array` and `Int32Array` views, managing little-endian header fields via `DataView` — is exactly where type safety prevents silent data corruption. Vite's TS support is zero-config.

## Future Work

1. [ ] **Replay Mode** — Load a simulation log file (migratory slice 307, complete) and replay it in the viewer with playback controls (play, pause, step, speed, scrub). The viewer becomes a debugging and analysis tool in addition to a live client. The log format uses the same binary serialization as the wire protocol. Dependencies: viewer slice 101 (deserialization). Effort: 3/5
2. [ ] **Interactive Controls** — Client-to-server commands: spawn/remove entities, adjust simulation parameters, pause/resume, change time scale. Requires migratory's "Administrative Client Commands" future work. Dependencies: Server-side command protocol. Effort: 3/5
3. [ ] **Day/Night and Seasonal Rendering** — Modulate lighting, fog color, and terrain/biome colors based on the server's temporal state (day/night cycle, seasonal parameter). Snow on terrain in winter, warm golden light at sunset, darker ambient at night. Dependencies: migratory slice 308. Effort: 2/5
4. [ ] **Multi-Resolution Entity Rendering** — Render aggregate entities (from migratory's future multi-resolution simulation) as cluster indicators rather than individual agents. A translucent sphere or icon showing count, heading, and dispersion for distant groups. Dependencies: migratory multi-resolution entity simulation. Effort: 3/5
5. [ ] **Recording and Export** — Capture the viewer's rendered output as video (via MediaRecorder API on the canvas) or export a snapshot as a high-resolution image. Useful for content creation and portfolio material. Effort: 2/5
