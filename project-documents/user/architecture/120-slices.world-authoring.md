# Slice Plan: World Authoring and Biome System

## Parent Document
[120-arch.world-authoring.md](120-arch.world-authoring.md) — Architecture for externalizing the viewer's visual configuration into versioned YAML tiers (profile, world, biome) with a strict-validation runtime loader.

## Planning Context
Architecture-level slice plan. The parent architecture document publishes an "Initial Slice Plan" summary, a "Pipeline shape across slices" table, and "Component Architecture Boundaries" section that together define what each slice does and does not touch. This plan ratifies that breakdown into the formal slice format. Slice numbers (121, 122, 123) follow the convention that slices in initiative N start at N+1; the parent architecture document and initiative plan have been updated in this same change to match.

Slice 121 is foundation work — the entire `src/config/` module, the manifest plugin, profile YAMLs, and the codebase migration to the `getConfig()` accessor. It must ship before 122 and 123 can begin. Slices 122 and 123 are feature slices that successively externalize biome and world content from `defaults.ts` into YAML, each leaving the viewer in a working state with renderer behavior unchanged.

There are no migration slices in the cross-slice sense — the *internal* migration of call sites from `import config` to `getConfig()` is bundled inside slice 121, because every consumer must change in the same diff for the system to remain working. There is no integration slice in this plan; integration concerns (per-cell biomes, hot reload, props/audio renderers) are deferred far-future and tracked under "Future Work."

## Foundation Work

1. [ ] **(121) Config Externalization Plumbing** _(design: [121-slice.config-externalization-plumbing.md](../slices/121-slice.config-externalization-plumbing.md))_ — New `src/config/` module per the parent's Component Architecture: `types.ts` (pure `ViewerConfig` and tier interfaces), `defaults.ts` (hardcoded fallbacks mirroring all current values from [src/config.ts](../../src/config.ts), including `DEFAULT_BIOME_NAME`), `schema.ts` (per-tier validators, `ATMOSPHERE_OVERRIDE_ALLOWLIST`, `HEX_COLOR_RE`), `loader.ts` (pure parse-validate-merge over raw YAML text — `parseProfile`/`parseWorld`/`parseBiome` returning `{ mergeable, resolution }`, plus `assemble()` running the merge in the order defined by the tier-composition ADR and a final structural check on the output), `fetcher.ts` (the only I/O module — manifest + tier fetches), `discovery.ts` (manifest parsing + name→path resolution), `selection.ts` (pure reader: URL `?profile=` → `localStorage` → `'default'`), `missing-texture.ts` (sentinel `Promise<Texture>` eager-loaded at `initializeConfig()`), `index.ts` (`initializeConfig()` idempotent with sticky errors, `getConfig()`, `getActiveBiomeName()`, test-only `resetConfig()`). Plus `viteConfigManifest` plugin generating `public/config/manifest.json` (`buildStart`+watcher in dev, `closeBundle` in build) scanning `profiles/`, `experiments/`, `worlds/`, and `biomes/`. Three checked-in profiles (`default`, `lowend`, `cinematic`) under `public/config/profiles/`. Strict per-tier schema validation (keyspace + structure + types + `schemaVersion: 1`). Codebase-wide migration from `import config` singleton to `getConfig()` accessor — every consumer in the same diff. Pipeline runs `defaults → profile.mergeable` only at this stage; world and biome content stays in `defaults.ts`. Dependencies: (110) and (111) shipped (current `ViewerConfig` shape mirrored into `defaults.ts`). Risk: Medium — accessor migration touches every existing config consumer; manifest plugin is new code. Effort: 3/5

## Feature Slices (in implementation order)

2. [ ] **(122) Minimum-Viable Biome Packs** — Establish the `public/biomes/<name>/` directory convention with a self-contained pack format (biome.yaml plus `textures/` directory, paths relative to the pack). Biome schema in `schema.ts`: tier-confined keyspace, reserved `props`/`audio`/`particles` slots that accept empty values and reject non-empty ones with a clear error. Two biomes shipped to prove genericity: `alien-vegetation` (migrated from the existing hardcoded biome that previously lived in `defaults.ts`) and one new biome (e.g. `desert-rock`) demonstrating that pack swapping requires no code change. The manifest plugin's biome scan begins populating the manifest's `biomes` map; `discovery.ts` resolves biome names to paths. `fetcher.ts` gains the biome fetch step (step 8 in the data flow), driven by `defaults.DEFAULT_BIOME_NAME` (no `world.yaml` exists yet). `loader.assemble()` runs the now-three-stage pipeline `defaults → biome.mergeable → profile.mergeable` for the biome's own keyspace (`surface`, `cliff`, `slopeBlend`); biome-level `atmosphereOverrides` is accepted by the schema (with empty-content semantics) but the unwrap merge stage is not yet active — that arrives in slice 123. Magenta-checker missing-texture sentinel ships under `public/assets/missing-texture.png`, eagerly loaded by `missing-texture.ts`; `Three.js` `TextureLoader.onError` substitutes the sentinel for any failed biome texture fetch with a console error naming the biome, field, and URL. DEV-mode HEAD preflight on biome textures emits warnings without blocking startup. Renderer behavior is unchanged — the same `surface`/`cliff` values that previously lived in `defaults.ts` now reach `getConfig()` via YAML. Dependencies: (121). Risk: Low — biome content moves to YAML but the consuming renderer code from (110)/(111) is untouched. Effort: 3/5

3. [ ] **(123) World Atmosphere Tier and Override Mechanism** — Extract sky / sun / fog (and `terrain.slabDepth`) from `defaults.ts` into `public/config/worlds/<name>.yaml`; `default.yaml` is checked in. The `defaultBiome` constant migrates from `defaults.DEFAULT_BIOME_NAME` to a resolution-only key on the world tier; `fetcher.ts` now resolves the biome name from `world.resolution.defaultBiome` instead of `defaults.ts`, and the world-fetch step (step 7 in the data flow) becomes active. World schema enforced via `schema.ts` validators sharing the same per-root definitions used for `atmosphereOverrides` leaf validation. Implements the full deep-merge pipeline: `defaults → world.mergeable → biome.mergeable → unwrap(biome.atmosphereOverrides) → profile.mergeable`, with the `atmosphereOverrides` extract-and-rebase step unwrapping into world-tier paths. Allowlist enforcement is active: `ATMOSPHERE_OVERRIDE_ALLOWLIST = ['sky', 'sun', 'fog']`. Biome validator runs the two-step override check — top-level key in allowlist, then `validatePartial(root, obj)` against the world schema's per-root definitions (only present fields type-checked, unknown fields hard-fail, required-field enforcement skipped). YAML `null` (`~`) deletion-marker semantics fully active in deep merge. Renderer behavior is unchanged for the default-world / default-biome combination; the second biome from slice 122 gains a non-empty `atmosphereOverrides` block as a real-world test of the override flow. Dependencies: (121), (122). Risk: Low — pipeline is additive, merge function from (121) accepts the new inputs without restructuring. Effort: 2/5

## Notes

- **Slice numbering follows the N+1 convention** — slices in initiative 120 start at 121. The parent architecture's "Pipeline shape across slices" table, "Component Architecture Boundaries" section, and the data flow's `[122+]`/`[123+]` annotations all use these same numbers. Slice 121 is the next slice in implementation order after slice 112 (Terrain Wire Protocol v2, shipped); the gap (113-120 unassigned) reflects that 121 is the first slice in a new initiative, not a continuation of the 100-series.
- **Each slice leaves the viewer in a working state.** The architecture's "Pipeline shape across slices" table is the authoritative description: same merge function across all three slices, applied to whichever inputs exist; `defaults.ts` shrinks as content moves into YAML. `getConfig()` returns the same effective values before and after each step (renderer behavior unchanged across all three slices).
- **Slice 121 is the largest because it owns the codebase-wide accessor migration.** The triple-role decomposition of the existing `src/config.ts` (types / data / accessor) plus the `import config` → `getConfig()` substitution at every call site cannot be split across slices — leaving a half-migrated codebase would create exactly the test-isolation and test-interdependence problems the architecture's "Why the module split" section is designed to prevent.
- **`atmosphereOverrides` schema slot ships in slice 122 but the merge step does not.** Slice 122's biome schema accepts the `atmosphereOverrides` key but rejects non-empty content (the override mechanism doesn't exist until slice 123). Slice 123 implements the deep-merge unwrap step and the allowlist enforcement.
- **No hot reload in any of the three slices.** The architecture's hot-reload non-goal ADR is binding: slices 121-123 must not write code paths that assume hot reload (no file-watcher subscriptions, no in-place rehydration). Page refresh is the supported path. Hot reload is owned by a future slice co-incident with the in-app biome-picker UI.
- **Strict failure semantics throughout.** Schema validation failures hard-fail at startup with a blocking error overlay (no degraded mode). Missing textures fall back to the magenta-checker sentinel with a console error. Unknown YAML fields anywhere are a hard error. Sentinel load failure is itself fatal — see the asset-loading-failures ADR.
- **No new external runtime dependencies beyond `js-yaml`** in any of the three slices. Future slices that ship 3D props or audio will introduce their own dependencies scoped to those slices.

## Future Work
Items identified during architecture review that are out of scope for slices 121-123. Numbering restarts at 1 per the slice-planning template.

1. [ ] **Per-Cell Biome Data and Edge Blending** — Server protocol extension declaring biome IDs per terrain cell, plus renderer multi-material blending at biome boundaries. Likely two slices: protocol/data on the receive side, renderer on the visual side. The 120-series file format is forward-compatible, but the renderer's consumption interface in `src/rendering/` will change (see "Future-compatibility scope" in the parent architecture document). Dependencies: 120-series complete; migratory server protocol extension. Effort: TBD
2. [ ] **3D Prop Assets** — Populates the reserved `props` schema slot in biome.yaml. Asset-loader slice; biome packs gain `props/` subdirectory. Schema is forward-compatible by construction (slot reserved by slice 122). Dependencies: 120-series complete. Effort: TBD
3. [ ] **Biome Audio Assets** — Populates the reserved `audio` schema slot. Audio-loader slice; biome packs gain `audio/` subdirectory. Forward-compatible by construction. Dependencies: 120-series complete. Effort: TBD
4. [ ] **Biome Particle Effects** — Populates the reserved `particles` schema slot. Particle-system slice; depends on the renderer's particle infrastructure (not yet in place). Forward-compatible by construction. Dependencies: 120-series complete; particle infrastructure. Effort: TBD
5. [ ] **Hot Reload of Config and Biome Assets** — Editing a YAML file or biome asset rehydrates the running viewer without a page refresh. Owns the open design questions deferred from the parent architecture's hot-reload non-goal: full-config rehydrate vs. tier-only, validation-failure fallback policy, immediate vs. on-transition revert when overrides are removed. Likely co-incident with the in-app biome-picker UI. Dependencies: 120-series complete; usage data on edit-refresh friction. Effort: TBD
6. [ ] **In-App Biome Picker UI** — Dropdown UI listing discovered biomes (drawn from the manifest) for runtime selection, with the choice persisted to the same `localStorage` key that slice 121's `index.ts` writes after manifest validation. Likely co-incident with hot reload (above). Dependencies: 120-series complete. Effort: TBD
7. [ ] **Schema Migration Helpers (v1 → v2)** — Tooling for `schemaVersion` bumps when the first breaking change lands (per-tier independent migration). Until then every file in the repo is v1 and no helpers are needed. Dependencies: first breaking schema change. Effort: TBD
